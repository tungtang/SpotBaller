Basketball video analytics

How detection models fit your pipeline, how to train or fine-tune them, which checkpoints help, and how that supports an app that turns game footage into per-player fundamentals.

1. How the model works (in your stack)

Your project centers on Ultralytics YOLOv8: a single-stage object detector that predicts bounding boxes and class labels in one forward pass per frame. The checkpoint models/e-bard/BODD_yolov8n_0001.pt comes from the E-BARD basketball detection family (Hugging Face: GabrieleGiudici/E-BARD-detection-models), trained for court-relevant objects rather than generic COCO classes.

What one frame looks like to the model
  • Input: a resized image tensor (training defaults use imgsz=1280 in your scripts).
  • Output: boxes, scores, and class IDs for objects the head was trained on.
  • Your dataset config (app/ml/yolo_data.yaml) defines classes: player, ball, rim, jersey_number — the deployed E-BARD weights may use a subset; align labels when you fine-tune.

Detection alone is not “stats.” The app turns boxes into analytics by chaining:

  1. Tracking (e.g. ByteTrack) — stable IDs across frames for each player and the ball.
  2. Identity — jersey OCR and team clustering so “player 7” maps to a real role on a team.
  3. Events — shot attempts, makes/misses, possession changes from geometry + ball proximity.
  4. Aggregation — minutes, touches, FGA/FGM, and team splits exported as JSON.

That matches your MVP spec: fixed-camera full-court or similar footage, with quality targets on detector mAP, track continuity, and stat error vs manual labels.

2. How you can train (or fine-tune)

Training uses the Ultralytics API. Your repo provides two entry points:

  • python -m app.ml.train_yolov8 — trains from a base checkpoint with --data app/ml/yolo_data.yaml, default --epochs 100, --imgsz 1280, output under runs/basketball/.
  • python -m app.ml.train_iteration --name iter1 --model models/e-bard/BODD_yolov8n_0001.pt — wraps an iteration workflow starting from E-BARD (see docs/model_training_plan_2weeks.md).

Prerequisites:

  • Labeled images in YOLO format and split manifests: data/splits/train.txt, val.txt, test.txt (paths referenced in yolo_data.yaml).
  • Python env with ultralytics installed (pip install -r requirements.txt per README).

Example direct training invocation (from project root):

python -m app.ml.train_yolov8 \
  --model models/e-bard/BODD_yolov8n_0001.pt \
  --data app/ml/yolo_data.yaml \
  --epochs 100 \
  --imgsz 1280

On Apple Silicon, train_yolov8.py uses device="mps". For Linux/NVIDIA, adjust device in the script or use Ultralytics’ default CUDA auto-selection.

After deployment, improve data iteratively: error_mining uses calibration reports to surface hard clips for re-labeling (see README “Model training workflow”).

3. Pre-trained models you can leverage

Layer these by responsibility — no single model delivers full player strengths/weaknesses; you combine specialized checkpoints.

Role What to use Notes
Player / ball / rim detection E-BARD BODD_yolov8n_0001.pt Basketball-tuned YOLOv8n; your default when file is present (weights=auto).
Generic YOLOv8 baseline yolov8n.pt … yolov8x.pt (Ultralytics) Fallback or transfer starting point; not sport-specific — fine-tune on your labels.
Multi-object tracking ByteTrack Stable IDs from detections; your pipeline defaults to ByteTrack with centroid fallback.
Jersey numbers PARSeq-style STR / sports OCR fine-tunes Tracklet voting + confidence filtering for stable numbers.
Team assignment SigLIP / CLIP embeddings + clustering Jersey appearance → team buckets (see player_identity_map.json in your stack).
Action verification (optional) VideoMAE / 3D CNN classifiers Helps verify passes, blocks, steals beyond geometry-only rules.

Hugging Face hosts many sport and pose models; for your repo’s first integration, the explicit shortlist names E-BARD detection models as the primary basketball detector.

4. End goal: upload games → fundamentals per player

Product shape

  • Upload MP4 from different teams/cameras (target: 1080p, ≥30 FPS, H.264).
  • Backend job: detect → track → identify → events → calibrated stats.
  • Outputs: per-player minutes, touches, FGA/FGM, possession-related signals; team rollups in team_box_score.json-style artifacts.
  • UI: FastAPI + Streamlit dashboard (already scaffolded) for review and iteration.

“Strengths & weaknesses” builds on fundamentals: shooting efficiency under volume, time on ball vs turnovers proxy, defensive events when you add action models — start with MVP stats, then layer shot charts and role tags as data quality improves.
