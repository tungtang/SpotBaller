SpotBaller Parallel Architecture Plan

Decision-first design doc for improving throughput while preserving tracker correctness and output parity.

1) Why Parallelize Carefully

The pipeline has temporal dependencies: tracker state at frame N depends on frame N-1. This means not every stage should be parallelized. We target independent or pipeline-friendly work first.

Safe first: decode prefetch, output writing, independent OCR tasks
Caution: tracker update order, GPU worker fan-out, memory pressure

2) Current Flow (Conceptual)

Read frame from video
↓
Preprocess + detect players/ball
↓
Update tracker (stateful, ordered)
↓
Optional pretrained stack (team/OCR/action hints)
↓
Render annotated frame + write outputs + aggregate stats

3) Proposed Parallel Target Architecture

A. Producer Thread

Decodes and preprocesses frames into a bounded queue.

I/O-bound
Low risk

B. Main Ordered Worker

Runs detection + tracker update in strict frame order.

Correctness-critical
No reordering

C. Optional Task Pool

Runs independent crop-level extras (OCR/action hints) with bounded workers.

Conditional
CPU/GPU dependent

D. Writer Thread

Handles annotated video writes + periodic JSON/stat flushes asynchronously.

Removes output stalls
Low risk

4) Rollout Phases and Gates

Phase Changes Expected Gain Go/No-Go Gate
Phase 1: Baseline Instrument stage timers and throughput metrics. Observability only Proceed only with clear bottleneck signal.
Phase 2: Low-Risk Parallel Add frame prefetch queue + async writer queue. ~15-30% typical No output regressions, stable memory.
Phase 3: Conditional Pool Parallelize independent extras (OCR/hints) when dominant. Workload-dependent Deterministic merge, bounded VRAM/CPU usage.
Phase 4: Hardening Add flags, backpressure, shutdown/drain semantics, stress tests. Reliability No deadlocks, clean cancellation, artifact completeness.

5) Guardrails

  • Tracker and core stats remain ordered by frame index.
  • Bound all queues to avoid unbounded RAM growth.
  • Feature-flag each parallel component for safe rollback.
  • Graceful stop: set stop event, drain queues, flush final artifacts.
  • Benchmark on /basketball_analysis/input_videos/video_1.mp4 for before/after comparison.

6) Decision Options

Conservative

Implement Phases 1-2 only.

Best balance of gain vs complexity.

Balanced

Implement Phases 1-3 with bounded extra-task pool.

Higher gain if OCR/extras are bottleneck.

Aggressive

Broader concurrency and GPU scheduling optimization.

Most complex, highest regression risk.
